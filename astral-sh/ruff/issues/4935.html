Don't ask me why, but I spent the last two days of my holiday going over all the Sonar rules.

Ruff has already implemented 85 of 311 currently active and relevant Sonar rules. I also found 15 rules where Sonar might have a better or more complete implementation. I am not sure how to d

When testing all rules, I found three broader categories of Sonar rules that are currently not implemented in Ruff:

  1. Regex: Sonar has 25 Regex related rules. I think these would be a great addition to Ruff, as regexes are notoriously unintuitive, and these rules help simplify and speed up some regexes
  2. AWS: Sonar has 18 AWS related rules. Personally I don't use AWS, so I can't vouch for the added value of these rules, but with the prevalance of AWS I think implementing it would help many codebases make 'dumb' security mistakes.
  3. Typing: Sonar is not a full type checker, but it has quite some rules that do check some types related logic. I am not sure if this is on the roadmap for Ruff (or if it is even technically possible), but I think it would be very cool if Ruff could also implement some of these type checks. Many repositories also use a full-fledged static type checker, but these are generally quite slow, so if Ruff can do a first check in a fraction of a second that would, in my opinion, be very beneficial for development speed.

Regarding rule sets. I am not sure if it makes a lot of sense to create a new Sonar ruleset, as Sonar does not check a single type of issue and many of the rules overlap with existing rules. I think it makes more sense to add the Sonar rules to existing rulesets (e.g., PD for Pandas related rules) or create a new rule set (e.g., REG or RGX for regex rules and AWS for AWS related rules).
