[ty] Always register rename provider if client doesn't support dynamic registration

#21789 opened by @MichaReiser on 2025-12-04 11:04
@MichaReiser
Summary

Always register the rename provider if the client doesn't support dynamic registrations so that users don't need to know that renaming support can only be configured using initialize_options.

Fixes astral-sh/ty#1745

Test Plan

I removed the dynamic registration check in server_capabilities and tested that symbol renaming isn't working in VS Code when disabling rename but is working when enabled. I also verified that VS Code uses the rename functionality of any other available LSP (it doesn't block renaming if one LSP returns None and an other returns Some).
