[RUF021]: Add an autofix

#9449 opened by @AlexWaygood on 2024-01-09 12:49
@AlexWaygood
Summary

This adds an autofix for the newly added RUF021 (see #9440).

The PR currently only adds an autofix to the diagnostic if:

  1. There are no line breaks inside the expression as it currently exists;
  2. We determine that we could safely add parentheses to the subexpression in question without the line getting too long.

If we know that both of the above are true, the fix is pretty simple to implement. I think it should also be pretty safe, as well -- we shouldn't be making any changes here that impact the AST.

Two remarks:

  1. It would be possible to add an autofix if the pre-existing expression spanned multiple lines, but it would be much harder, I think. You'd have to worry about whether there were comments at the end of lines or in between lines; you'd have to worry about indentation; you'd have to worry about whether the newly parenthesized subexpression should be formatted into a single line or should itself be split into several lines. (But maybe there's an easy way of resolving all those things that I'm not aware of üòÑ.)
  2. Currently the PR just doesn't add a fix if the resulting fix would cause the line to be too long. But maybe we could add the fix even then, but just mark it as unsafe? Not sure.

This operation compares the byte-length instead of the text width. The byte and text width are identical for ASCII characters but non-ASCII characters either require multiple-bytes to represent them. Then there are Emojis and east-asian characters that take multiple columns when rendered on screen. Last, there's also the tab character which uses up to tab-width columns when shown in the editor.

Luckily, Ruff has the infrastructure to compute the visual width of a line. Take a look at the LineWidthBuilder and its usages to compute the width of a line.

https://github.com/astral-sh/ruff/blob/84979f96736aefca6d19f656e68600a92690c737/crates/ruff_linter/src/line_width.rs#L125-L133

I would suggest using usize::from(line_length) here. They are equivalent in this context because a u16 will always fit into a usize, but as casts are something of an annoyance in Rust-land. (Because in general they do silent truncation. And... other things.)

Also, I'm not sure what the acceptable bounds of checker.settings.line_length are. IIRC, its minimum value is 1, so subtracing 2 could result in an underflow. For that, I think I'd suggest saturating arithmetic.

So, that's usize::from(line_length).saturating_sub(2).

(Although I now see Micha's comment, so this code might need to change to be a little more complicated.)
