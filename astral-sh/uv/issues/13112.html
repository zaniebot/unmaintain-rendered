SHA mismatch if wheel has no sha in the index, but sdist has

#13112 opened by @f3flight on 2025-04-26 01:37
@f3flight
Summary

We have an internal simple index API (html) at the company, and due to some issues some artifacts don't have #sha256=... suffix in the link. Specifically, some packages have sha in sdist link, but not in wheel link. When UV creates a lockfile, it won't put sha for these wheels into the lockfile, and will write only sha for sdist, reflecting the state of the index.

If I check the lockfile, it looks like this:

[[package]]
name = "somepackage"
version = "1.0.19"
source = { registry = "https://internal-index" }
dependencies = ["...redacted..."]
sdist = { url = "https://internal-storage/somepackage-1.0.19.tar.gz", hash = "sha256:9e7a94344c6fb4f792eaa7c881693af0edc157e88c497e2606efd94ad7d0c686" }
wheels = [
    { url = "https://internal-storage/somepackage-1.0.19-py3-none-any.whl" },
]

However, when trying to uv sync, uv will fail like so:

(venv310) user@pc [ ~/repos/somerepo ]$ uv sync --active --locked --group test
Resolved 377 packages in 4ms
  × Failed to download `somepackage==1.0.19`
  ╰─▶ Hash mismatch for `somepackage==1.0.19`

      Expected:
        sha256:9e7a94344c6fb4f792eaa7c881693af0edc157e88c497e2606efd94ad7d0c686

      Computed:
        sha256:79b282a73f3d5717191011ffdb7d94d9cebb4942f82f288eba8ea66721ae5203

When I used -v for more debug, I saw that uv downloads the wheel. Looks like uv tries to use SHA from sdist when verifying downloaded wheel file:

(venv310) user@pc [ ~/repos/somerepo ]$ curl -s https://internal-storage/somepackage-1.0.19-py3-none-any.whl | sha256sum
79b282a73f3d5717191011ffdb7d94d9cebb4942f82f288eba8ea66721ae5203  -

Sounds like a bug!
