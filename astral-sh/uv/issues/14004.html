Summary

The best idea I've been able to come up with is to add some kind of a wheel regex filter at lock time to handle filtering on manylinux specifics, etc. since there aren't available markers / constraints for this afaict

In my scenario we use sam for python lambda deploys, which requires a requirements.txt which we create via uv export, and then sam hard codes manylinux2014 (see https://github.com/aws/aws-sam-cli/issues/7860 and https://github.com/aws/aws-sam-cli/issues/7744) but that causes downstream issues, so we're looking to create a requirements.txt (and ostensibly keep uv.lock in sync) which only contains versions of packages which have wheels build for manylinux_x_y, for instance.

--filter-wheels manylinux_2_28 or something like that? I'm not sure.
