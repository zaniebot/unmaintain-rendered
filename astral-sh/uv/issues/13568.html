Resolution failure for tensorflow on macos

Closed #13568 opened by @GuillaumeDehaene on 2025-05-21 09:12
Issue opened by @GuillaumeDehaene on 2025-05-21 09:12

Solution

Adding platform information in the required-environments fixes the issue

pyproject.toml

[tool.uv]
# Require that the package is available for macOS ARM
required-environments = [
    "sys_platform == 'darwin' and platform_machine == 'arm64'",
]

Summary

I have an example of a minimal state which produces a resolution failure for tensorflow on platform macosx_15_0_arm64 Adding tensorflow as a direct add fixes the failure. I believe this indicates that there would be a valid resolution which is somehow not found from the initial requirements.

I am a very novice uv user so it is very possible that I am misunderstanding some of the details

Minimal example

  • python version 3.10.17
  • platform: macosx_15_0_arm64

Trying to add packages protobuf and retina-face simultaneously or in the following order produces a resolution failure:

Failure commands:

>>> uv add protobuf retina-face

>>> uv add protobuf && uv add retina-face

error message:

Resolved 53 packages in 289ms
error: Distribution `tensorflow==2.9.0 @ registry+https://pypi.org/simple` can't be installed because it doesn't have a source distribution or wheel for the current platform

hint: You're on macOS (`macosx_15_0_arm64`), but `tensorflow` (v2.9.0) only has wheels for the following platforms: `manylinux_2_17_x86_64`, `manylinux2014_x86_64`, `macosx_10_14_x86_64`, `win_amd64`

The following modifications do not produce a resolution error:

  • adding sequentially in the reverse order: retina-face then protobuf
  • adding tensorflow to the requirements / to the initial simultaneous add

Correct commands

>>> uv add retina-face && uv add protobuf

>>> uv add tensorflow && uv add protobuf retina-face

>>> uv add tensorflow protobuf retina-face
