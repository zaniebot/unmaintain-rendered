"Project API" equivalent to `uv pip list`?

Closed #7006 opened by @matterhorn103 on 2024-09-04 11:29
Issue opened by @matterhorn103 on 2024-09-04 11:29

Maybe I am the only one who ever does this, but especially when it comes to non-package projects I often like to reassess the dependencies of my project.

At the moment this can be done with uv pip list, but that a) shows the currently installed packages, not the dependencies as listed in uv.lock or pyproject.toml, and therefore b) doesn't show the constraints that are placed on the dependencies but rather the currently installed versions, and c) is part of the pip compatibility API.

Naturally just inspecting pyproject.toml is an option, but that kind of argument could be used to argue against the need for uv add/uv remove as well.

Is an equivalent uv list or something planned for the "project" API? Or should I expect to continue using uv pip list for the foreseeable future?

(It is of course also common to use pip list with flags for other purposes like checking for outdated packages.)

By the way, just wanted to at the same time express my gratitude for the changes you introduced in uv 0.4! Having first-class support for non-package projects in this way, along with workspaces, has really improved its usability for scientific work, really makes uv stand out as a potentially universal tool, and it is really appreciated that you focussed on such use-cases even before the initial stable release. :)
