# authentication for two private package registries: unexpected urls

---

_Issue opened by @dd-ssc on 2025-03-06 13:25_

## TL;DR

We managed to make some progress in getting authentication for private gitlab.com package registries to work with `keyring` and a forked, heavily modified 1Password backend, but now we're stuck and confused about the weird urls `uv` is requesting credentials for: It seems as if `uv` "mixes" index urls and package names.

## The Problem

Creating a venv for a project with one dependency on a package from a private gitlab package registry works ‚úÖ.
Creating a venv for a project with two dependencies on packages from two private gitlab package registries fails ‚ùå.


## The Keyring 1Password backend

This is beyond the scope of the uv project, but some wider context might be helpful.

<details>
  <summary>Details</summary>

We tried to use [Keyring](https://github.com/jaraco/keyring) with the 1Password third party backend mentioned on the keyring page - [`onepassword-keyring`](https://github.com/falling-springs/onepassword-keyring) - but that implementation was apparently never meant for anything more than personal use by its author. It is currently dysfunctional - it never returns any credentials, but only a `[use 'op item get <item ID> --reveal' to reveal]` message - unless its source code is monkey-patched.

There are [two public forks](https://github.com/falling-springs/onepassword-keyring/forks), one of which [fixed the problem mentioned](https://github.com/rdhyee/onepassword-keyring/commit/b9190a47810e070b72bba55bc25b944f8b03c61a), but never published that fix as a package.

The alternative we are currently using is [`keyrings.onepassword`](https://github.com/risqcapital/keyrings.onepassword); we forked that project and added some logging to a file: `keyring` and its backends communicate over `stdout`, so can't just use `print(...)` for logging.

As `uv` works with _index urls_, but 1Password works with _vault items_, we also added a rudimentary map to translate the former to the latter.

Finally, to reduce the effort to reproduce this issue to the least amount possible, we removed the need to install the 1Password application, the 1Password CLI, set up any items in any 1Password vaults or pass any environment variables.

This is just an awful hack. But maybe it helps us to fix this issue.

See <https://github.com/DesmoDyne/keyrings.onepassword> for details.
</details>

## Further questions and concerns

1. Why does `uv` append any package name to the index url configured in `pyproject.toml` at all ? We've been using package registry urls ending in `.../simple` for many years (but then - we always had our packages on gitlab.com ü§∑üèª‚Äç‚ôÇÔ∏è) - which is also really similar to the [PyPI base url](https://packaging.python.org/en/latest/specifications/simple-repository-api/#base-html-api) `https://pypi.org/simple/`; possibly related: [11236](https://github.com/astral-sh/uv/issues/11236).
2. Why is there a query for `https://pypi.org/simple/uv-mre-01-dep-02/` ? The [Index docs](https://docs.astral.sh/uv/concepts/projects/dependencies/#index) say
   > Using an `index` source pins a package to the given index ‚Äî it will not be downloaded from other indexes.

   but from our observations, this doesn't seem true ü§∑üèª‚Äç‚ôÇÔ∏è We also tried adding `explicit = true` to the `[[tool.uv.index]]` sections, taken from [Pinning a package to an index](https://docs.astral.sh/uv/configuration/indexes/#pinning-a-package-to-an-index), but that doesn't seem to do anything ü§∑üèª‚Äç‚ôÇÔ∏èü§∑üèª‚Äç‚ôÇÔ∏èü§∑üèª‚Äç‚ôÇÔ∏è
3. Does the query for `https://pypi.org/simple/uv-mre-01-dep-02/` mean that a package with the same name is installed from PyPI - instead of from our own package registry - if authentication fails ?
4. Do I have to be worried that that query leaks the credentials for our package registry to PyPI ?
