### Rust Version

rustc 1.90.0 (1159e78c4 2025-09-14)

### Clap Version

4.5.50

### Minimal reproducible code

`src/main.rs`:
```rust
use clap::Parser;

#[derive(clap::Parser, Debug)]
struct Cli {
    #[arg(long)]
    /// comment style doc
    ///
    #[doc = "attr style doc"]
    #[doc = include_str!("test.md")]
    flag: String
}

fn main() {
    let _cli = Cli::parse();
    println!("no-op");
}
```

`src/test.md`:
```
include style doc
```


### Steps to reproduce the bug with the above code

cargo r -- --help

### Actual Behaviour

It outputs this:
```
Usage: example --flag <FLAG>

Options:
      --flag <FLAG>
          comment style doc

          attr style doc

  -h, --help
          Print help (see a summary with '-h')
```

### Expected Behaviour

Instead it should output this:
```
Usage: example --flag <FLAG>

Options:
      --flag <FLAG>
          comment style doc

          attr style doc
          include style doc

  -h, --help
          Print help (see a summary with '-h')
```

I found maybe not so related issue in rust-analyzer repo: https://github.com/rust-lang/rust-analyzer/issues/11137 and in axum: https://github.com/tokio-rs/axum/issues/929

_Comment by @epage on 2025-10-22 15:28_

Looks like we are only looking for `#[doc = <string-literal>]` attributes, see
https://github.com/clap-rs/clap/blob/df7bdfc4996ba2f4388e2c01a6ab4cbe26b2e4df/clap_derive/src/utils/doc_comments.rs#L14-L31

The big problem is if we're getting `include_str` pre-expansion which it looks like we are.  We'd have to re-emit that and can't do any processing on it.  We either then need a way to concatenate the strings at compile time *or* we condition support on this on the `string` feature.

```
Timelike value has the following syntax:
<empty> - now
YYYY-MM-DD HH:MM - explicit set timestamp
@HH:MM - current day with set time
-<dur> - current time minus <dur> offset
+<dur> - current time plus <dur> offset

where <dur> can be anything that `humantime` can parse (example: 2d5h29s)
```

_Comment by @epage on 2025-10-22 19:14_

The pre-processing is primarily to split out short help from long help.  With the builder API, users might as well specify them separately (and its a lower level API).  So this is important for the derive API and not fitting within the builder API.  Moving it to runtime would also require people using the `string` feature which slows down compile times, bloats the binary, and likely slows things down.
