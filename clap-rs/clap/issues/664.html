args_from_usage() has issues with non-ASCII identifiers

#664 opened by @tormol on 2016-09-15 21:33
@tormol

Non-ASCII help on its own is OK: clap::App::new("").args_from_usage("<h> 'with ŭnicode'"); works

If the last character is non-ascii there are slice errors: clap::App::new("").args_from_usage("<hæ> 'comment doesn't matter'");

If both name and help has non-ascii the program hangs: clap::App::new("").args_from_usage("<øh> 'with ŭnicode'");

If only the name is non-ascii the argument is not detected: clap::App::new("").args_from_usage("<øh> 'ASCII or empty'").get_matches().value_of("øh").unwrap()

The first three act the same with options, but this is different: clap::App::new("").args_from_usage("--øh 'ASCII'").get_matches(); gives this error:

error: Found argument '--øh' which wasn't expected, or isn't valid in this context
    Did you mean --ø?

USAGE:
    clapbug --ø <ASCII'>

For more information try --help

Arg::with_name() has none of these issues.
